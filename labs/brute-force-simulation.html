Simulating Brute Force Attacks: Real-World SOC Detection Lab

Authentication attacks are one of the most common initial access methods used by threat actors. From nation-state APTs to ransomware operators, attackers consistently target authentication mechanisms to gain their first foothold in enterprise networks.

In this lab exercise, I simulate a brute force attack against my Active Directory environment, configure enterprise-grade defensive controls, and demonstrate how a SOC analyst detects, analyzes, and responds to credential-based attacks.

This isn't just about running an attack tool. It's about understanding the complete security lifecycle: threat simulation → defensive controls → detection → investigation → response.

The best way to understand how attacks appear in logs is to simulate them in a controlled environment. This is exactly what red team exercises, purple team operations, and security validation programs do in enterprise environments.

Why Brute Force Attacks Matter

Brute force attacks remain relevant in 2026 despite advances in authentication technology. Understanding why helps explain why SOCs still prioritize detection of these attacks.

Real-World Impact:

  • 81% of data breaches involve compromised credentials (Verizon DBIR)
  • Password attacks increased 74% year-over-year (Microsoft Security Report)
  • Average of 921 attacks per second against cloud services (Microsoft)
  • Ransomware operators frequently use brute force for initial access

Why Attackers Still Use Brute Force:

  • Human Factor: Users still choose weak or reused passwords
  • Legacy Systems: Not all systems support MFA or modern authentication
  • Success Rate: Even a 1% success rate across thousands of accounts provides access
  • Low Cost: Automated tools make brute force attempts cheap and scalable
  • VPN/RDP Exposure: Remote access services are common brute force targets
SOC Perspective: Even organizations with strong password policies and MFA still see brute force attempts. Detection and response capabilities must exist regardless of preventive controls, because no control is 100% effective and legacy systems always exist.

MITRE ATT&CK Framework Mapping

Before executing the simulation, I mapped the attack to the MITRE ATT&CK framework to understand what I should expect to detect.

Primary Technique:

Technique ID: T1110.001
Technique Name: Brute Force: Password Guessing
Tactic: Credential Access
Description: Adversaries with no prior knowledge of legitimate credentials attempt to guess passwords using common password dictionaries, lists of leaked credentials, or automated tools.

Related Techniques:

  • T1110 - Brute Force (parent technique)
  • T1110.003 - Brute Force: Password Spraying (alternative approach)
  • T1078 - Valid Accounts (post-compromise if successful)
  • T1021.002 - Remote Services: SMB/Windows Admin Shares (attack vector)

Detection Data Sources (Per MITRE):

  • Logon Session: Logon Session Creation (Event ID 4624/4625)
  • User Account: User Account Authentication (authentication logs)
  • Application Log: Application Log Content (failed authentication attempts)
Career Relevance: Enterprise SOC analysts are expected to map detected activity to MITRE ATT&CK. This standardized framework enables consistent communication across security teams and integration with threat intelligence platforms.

Defensive Control: Account Lockout Policy

Before simulating the attack, I needed to configure the same defensive controls that exist in enterprise environments. Account lockout policies are a fundamental defense against brute force attacks.

What is an Account Lockout Policy?

An account lockout policy automatically disables a user account after a specified number of failed authentication attempts. This prevents unlimited password guessing and forces attackers to choose between:

  • Speed: Attack quickly but trigger lockouts and alerts
  • Stealth: Attack slowly to avoid lockouts but take much longer

Configuration on DC01:

I configured the lockout policy through Group Policy Management, which is how enterprise environments deploy these controls across entire domains.

The Group Policy Management Editor shows the configured lockout policy:

Policy Settings:

Policy Setting Purpose
Account lockout threshold 5 invalid logon attempts Locks account after 5 failed passwords
Account lockout duration 15 minutes Account stays locked for 15 minutes
Reset account lockout counter after 15 minutes Failed attempt counter resets after 15 minutes

Applying the Policy:

After configuring the policy, I forced an immediate update across the domain:

C:\Users\Administrator> gpupdate /force Updating policy... Computer Policy update has completed successfully. User Policy update has completed successfully.

The successful policy update confirms that all domain-joined systems (including HR-PC01 and the Domain Controller) now enforce the lockout policy.

Enterprise Context: In production environments, Group Policy updates propagate automatically every 90 minutes (by default). Security teams use gpupdate /force for immediate policy enforcement after security configuration changes.

Attack Execution: SMB Brute Force with Kali Linux

To simulate a realistic attack, I used Kali Linux—a penetration testing distribution commonly used by both security professionals and attackers.

Why Kali Linux?

  • Pre-installed with professional penetration testing tools
  • Same tools actual attackers use (Hydra, Metasploit, Nmap, etc.)
  • Realistic attack simulation (not just Windows scripts)
  • Demonstrates cross-platform attack scenarios
  • Industry-standard for security testing

Attack Method: SMB Credential Brute Force

I targeted the SMB (Server Message Block) protocol, which is used for Windows file sharing and remote administration. SMB is a common attack vector because:

  • It's enabled by default on Windows systems
  • It requires authentication
  • It's commonly exposed to networks
  • Successful SMB authentication can enable lateral movement

Attack Command:

for i in {1..6}; do smbclient -L //192.168.10.10 -U testuser%WrongPass; done

Command Breakdown:

  • for i in {1..6} - Loop 6 times (exceeds 5-attempt threshold)
  • smbclient - SMB client tool
  • -L //192.168.10.10 - Target DC01 (Domain Controller)
  • -U testuser%WrongPass - Username and incorrect password
  • done - End loop

Attack Results:

The Kali Linux terminal shows the progression of the attack:

session setup failed: NT_STATUS_LOGON_FAILURE
session setup failed: NT_STATUS_LOGON_FAILURE
session setup failed: NT_STATUS_LOGON_FAILURE
session setup failed: NT_STATUS_LOGON_FAILURE
session setup failed: NT_STATUS_LOGON_FAILURE
session setup failed: NT_STATUS_ACCOUNT_LOCKED_OUT

Key Observations:

  • Attempts 1-5: NT_STATUS_LOGON_FAILURE - Authentication failed due to wrong password
  • Attempt 6: NT_STATUS_ACCOUNT_LOCKED_OUT - Account was automatically locked after 5 failures
Defensive Control Validated: The account lockout policy worked as designed. After 5 failed authentication attempts, the account was automatically disabled, preventing further brute force attempts.

SOC Alert Classification: Event ID 4740 (Account Lockout) is classified as a high-severity alert in most SOC environments. It requires immediate investigation because it could indicate either a brute force attack or a compromised account being used by an attacker who doesn't know the password.

SOC Level 1 Investigation Workflow

When a SOC Level 1 analyst receives these alerts, they follow a structured investigation process to determine severity and appropriate response.

Step 1: Initial Alert Triage

First, the analyst gathers basic context from the alert:

  • What happened? Account lockout due to multiple failed authentications
  • Which account? [email protected]
  • Which system? DC01 (Domain Controller) - high-value target
  • When? Mar 4, 2026 @ 15:31:58 UTC
  • How many attempts? 5+ within the same second
  • Source? Requires deeper investigation of event details

Step 2: Pattern Recognition

The analyst looks for indicators of automated vs. manual attacks:

Indicator Observation Interpretation
Timing Multiple failures within 1 second Automated tool (humans can't type that fast)
Consistency Regular intervals Scripted or tool-based attack
Target Single user account Password guessing (T1110.001)
Protocol SMB authentication Network-based attack

Step 3: Contextual Analysis

The analyst asks investigative questions:

  • Is the user aware of the failed logins? Contact user via phone/email/Slack
  • Is this expected behavior? Check for scheduled tasks, maintenance windows, or legitimate automation
  • What is the source IP? Internal network (192.168.x.x) or external?
  • Has this account been targeted before? Check historical alerts
  • Are other accounts being targeted? Look for similar patterns across other users
  • Was there any successful login? Check for Event ID 4624 from same source

Step 4: MITRE ATT&CK Mapping

The analyst maps observed behavior to the MITRE ATT&CK framework:

Primary Technique: T1110 - Brute Force
Sub-Technique: T1110.001 - Brute Force: Password Guessing
Tactic: Credential Access
Detection: Event ID 4625 (Failed Logon), Event ID 4740 (Account Lockout)

Step 5: Severity Assessment

L1 Severity Classification:

This Scenario: HIGH SEVERITY

Justification:
- ✅ Multiple failed authentications (5+)
- ✅ Automated attack pattern (sub-second intervals)
- ✅ Account lockout triggered
- ✅ Domain Controller targeted (high-value asset)
- ✅ Matches known attack technique (T1110.001)

Step 6: Response Actions

Based on the high-severity classification, the L1 analyst takes immediate action:

Immediate Actions:

  • ✅ Document findings in incident tracking system
  • ✅ Contact user to verify they were not attempting to log in
  • ✅ Check for successful logins from the same source IP
  • ✅ Review firewall logs to identify source IP and determine if it's internal or external
  • ✅ Search for similar patterns across other user accounts
  • ✅ Verify account is still locked (prevent attacker from resuming)

Escalation to Level 2:

This incident should be escalated to L2 SOC analysts for the following reasons:

  • High-Value Target: Domain Controller is critical infrastructure
  • Automated Attack: Indicates sophisticated tooling or persistent attacker
  • Unknown Source: Source IP needs investigation and potential blocking
  • Potential Ongoing Threat: Attacker may resume after account unlock
  • Forensic Analysis Needed: Determine if attacker gained access to other accounts

Defensive Recommendations

Based on this exercise, here are the defensive measures that would strengthen protection against brute force attacks in a real enterprise environment:

1. Multi-Factor Authentication (MFA)

MFA would have completely prevented this attack, even with a correct password. Modern enterprises deploy MFA for:

  • VPN access
  • Remote Desktop (RDP)
  • Email and Office 365
  • Administrative accounts
  • Cloud applications

2. Account Lockout Policy Tuning

My 5-attempt threshold was relatively permissive. Enterprise environments often use:

  • 3 attempts for privileged accounts
  • 5 attempts for standard users
  • Longer lockout durations (30-60 minutes) for critical systems

3. Network Segmentation

Limiting which systems can communicate with Domain Controllers reduces attack surface:

  • Restrict SMB (445/tcp) to authorized systems only
  • Firewall rules blocking external SMB access
  • VLANs separating user workstations from servers

4. Enhanced Monitoring and Alerting

SIEM rules should trigger alerts for:

  • 5+ failed logins within 5 minutes
  • Any account lockout event (Event ID 4740)
  • Failed authentications from unusual source IPs
  • Multiple accounts targeted from same source

5. Password Policies

  • Minimum 12-14 character passwords
  • Complexity requirements (uppercase, lowercase, numbers, symbols)
  • Password history (prevent reuse)
  • Regular password changes for privileged accounts
  • Blocklist of common passwords
