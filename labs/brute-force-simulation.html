Understanding Brute Force Attacks

Brute force attacks attempt to gain unauthorized access by systematically trying multiple passwords against a user account until finding the correct one.

MITRE ATT&CK: T1110.001

Technique: Brute Force: Password Guessing Tactic: Credential Access Description: Adversaries try many passwords against an account Detection: Failed authentication logs, account lockouts

Why Attackers Use This:

  • Targets weak or common passwords
  • No vulnerability exploitation required
  • Works against any authentication system
  • Can be automated with tools like Hydra, Medusa

Defensive Configuration: Account Lockout Policy

Account lockout policies prevent brute force attacks by temporarily disabling accounts after too many failed login attempts.

Group Policy Configuration on DC01:

Setting Configuration Purpose
Account lockout threshold 5 invalid attempts Lock account after 5 failed logins
Account lockout duration 15 minutes Account stays locked for 15 minutes
Reset counter after 15 minutes Failed attempt counter resets after 15 min
Group Policy Path: Computer Configuration → Windows Settings → Security Settings → Account Policies → Account Lockout Policy

Applying Group Policy:

gpupdate /force Updating policy... Computer Policy update has completed successfully. User Policy update has completed successfully.

Why gpupdate /force?

  • Forces immediate policy application
  • No need to wait for automatic refresh (90 minutes)
  • Confirms policy pushed to all domain computers
  • Essential for testing security controls immediately

Attack Execution: Kali Linux Brute Force

To validate the lockout policy, I simulated a brute force attack using Kali Linux targeting the testuser account.

Attack Command:

for i in {1..6}; do smbclient -L //192.168.10.10 -U testuser%WrongPassword done

Command Breakdown:

  • for i in {1..6} - Loop 6 times
  • smbclient -L - SMB client, list shares
  • //192.168.10.10 - Target DC01 (Domain Controller)
  • -U testuser%WrongPassword - Wrong password on purpose

Attack Results:

Attempt 1: NT_STATUS_LOGON_FAILURE Attempt 2: NT_STATUS_LOGON_FAILURE Attempt 3: NT_STATUS_LOGON_FAILURE Attempt 4: NT_STATUS_LOGON_FAILURE Attempt 5: NT_STATUS_LOGON_FAILURE Attempt 6: NT_STATUS_ACCOUNT_LOCKED_OUT
Defense Validated: Account lockout triggered after exactly 5 failed attempts as configured. Sixth attempt failed with NT_STATUS_ACCOUNT_LOCKED_OUT - proving security control is working.

Detection: Wazuh SIEM Alerts

Wazuh Alerts Generated:

Alert Rule ID Level Event ID Description
Failed logon attempts 60122 5 4625 Multiple authentication failures
Account locked out 60115 9 4740 Account lockout triggered

Alert Timeline:

  1. First 5 attempts: Wazuh Rule 60122 fires (Event ID 4625)
  2. Lockout triggered: Wazuh Rule 60115 fires (Event ID 4740)
  3. Sixth attempt: Another 4625 event with "account locked" status

Defense Effectiveness Analysis

What Worked:

  • ✅ Account lockout triggered after 5 attempts (as configured)
  • ✅ SIEM alerts generated immediately
  • ✅ Clear evidence trail in Windows Event logs
  • ✅ Attack successfully blocked

Limitations of Account Lockout:

  • ⚠️ Doesn't prevent password spraying (1-2 attempts per account)
  • ⚠️ Can be used for DoS (lock out legitimate users)
  • ⚠️ Temporary lockout - attacker can try again after 15 minutes
  • ⚠️ No protection if correct password is in first 5 attempts
Key Insight: Account lockout is ONE layer of defense. Effective security requires defense-in-depth: lockout policies + MFA + password complexity + monitoring + user education.
