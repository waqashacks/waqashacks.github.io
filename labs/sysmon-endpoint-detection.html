Executive Summary

What I Did: Deployed Sysmon (System Monitor) on Windows endpoint HR-PC01, configured advanced logging with SwiftOnSecurity config, and integrated with Wazuh SIEM for detecting suspicious PowerShell execution and process-level threats.

Why It Matters: Standard Windows event logs miss critical attack indicators like process creation, network connections, and file modifications. Sysmon provides EDR-level visibility that SOC analysts need to detect modern threats before they cause damage.

Result: Successfully deployed enterprise-grade endpoint monitoring, detected obfuscated PowerShell execution, and demonstrated ability to configure and operate endpoint detection tools used in production SOC environments.

Why Sysmon for SOC Operations?

Windows Security Logs Limitations:

  • ❌ No process command-line logging
  • ❌ Limited network connection visibility
  • ❌ No file creation monitoring
  • ❌ Minimal process relationship data

Sysmon Advantages:

  • ✅ Complete process execution details (Event ID 1)
  • ✅ Network connection logging (Event ID 3)
  • ✅ File creation monitoring (Event ID 11)
  • ✅ Registry modifications (Event ID 13)
  • ✅ Driver loading detection (Event ID 6)
  • ✅ Free, Microsoft-signed, enterprise-ready

Sysmon Deployment on HR-PC01

Step 1: Download Sysmon

Downloaded Sysmon v15.15 from Microsoft Sysinternals and extracted to dedicated folder.

Download URL: https://learn.microsoft.com/en-us/sysinternals/downloads/sysmon Version: v15.15 Extraction Path: C:\Users\Administrator\Downloads\Sysmon\

Step 2: SwiftOnSecurity Configuration

Implemented industry-standard Sysmon configuration from SwiftOnSecurity GitHub repository for comprehensive threat detection.

Config File: sysmonconfig-export.xml Source: SwiftOnSecurity Sysmon-Config Purpose: Balanced detection without overwhelming log volume Key Features: PowerShell logging, process creation, network connections

Why SwiftOnSecurity Config?

  • Industry-standard, battle-tested in enterprise SOCs
  • Balances detection coverage with log volume
  • Excludes noisy legitimate processes
  • Maintained by security community

Step 3: Installation Command

Sysmon64.exe -accepteula -i sysmonconfig-export.xml

Command Breakdown:

  • Sysmon64.exe - 64-bit Sysmon executable
  • -accepteula - Accept license agreement (required for automation)
  • -i - Install with specified config file
  • sysmonconfig-export.xml - SwiftOnSecurity configuration
Installation Success: Sysmon installed successfully and running as Windows service with advanced logging configuration.

Step 4: Service Verification

Verified Sysmon service is running and configured correctly.

sc query sysmon64 OUTPUT: SERVICE_NAME: sysmon64 STATE: 4 RUNNING
Service Validation: STATE: 4 RUNNING confirms Sysmon driver is loaded and actively monitoring endpoint activity.

Event Viewer Configuration

Sysmon Event Log Location:

Event Viewer Path: Applications and Services Logs → Microsoft → Windows → Sysmon → Operational

Sysmon writes events to a dedicated event log separate from Windows Security log, providing clean separation for analysis.

Sysmon Event ID 1: Process Creation

Event ID 1 is the most valuable Sysmon event for threat detection - it captures every process created on the endpoint with full details.

Critical Fields for Analysis:

Field Description Detection Value
CommandLine Full command executed Detect obfuscation, malicious flags
ParentImage Process that spawned this one Identify suspicious parent-child relationships
User Account that ran process Detect privilege escalation
Hashes File hashes (MD5, SHA256) Threat intelligence matching
Image Executable path Identify LOLBins, unusual locations

Threat Detection: Suspicious PowerShell Execution

To validate Sysmon detection capabilities, I executed obfuscated PowerShell command commonly used by attackers.

Attack Simulation:

powershell -nop -w hidden -encodedcommand [Base64String]

Suspicious Indicators:

  • -nop (NoProfile) - Bypass PowerShell profiles
  • -w hidden (WindowStyle Hidden) - Hide execution window
  • -encodedcommand - Obfuscated payload in Base64
Red Flag: PowerShell with hidden window + no profile + encoded command is a classic indicator of malicious activity. Legitimate scripts rarely combine all three flags.

Wazuh SIEM Detection

Sysmon events forwarded to Wazuh SIEM triggered multiple high-severity alerts.

Wazuh Alerts Generated:

Alert Level MITRE ATT&CK Description
Ingress Tool Transfer 15 T1105 Potential file download or tool transfer
Event Triggered Execution 12 T1546.011 Suspicious execution trigger mechanism
PowerShell Execution 4 T1059.001 PowerShell script detected
Windows Command Shell 4 T1059.003 Command-line interpreter usage
SOC L1 Analysis: Level 15 alert (Ingress Tool Transfer) requires immediate investigation. Combined with hidden PowerShell execution, this indicates potential compromise or malware activity.

SOC Investigation Workflow

Step 1: Alert Triage

  • ✅ Highest severity: Level 15 (T1105 - Ingress Tool Transfer)
  • ✅ Source: HR-PC01 endpoint
  • ✅ User: THREATWATCH\Administrator
  • ✅ Process: powershell.exe with suspicious flags

Step 2: Process Analysis

CommandLine: powershell -nop -w hidden -encodedcommand... ParentImage: C:\Windows\System32\cmd.exe User: THREATWATCH\Administrator Image: C:\Windows\System32\WindowsPowerShell\v1.0\powershell.exe

Red Flags:

  • Hidden window execution
  • Base64 encoded payload
  • NoProfile flag (bypass logging)
  • Spawned from cmd.exe (potential scripted attack)

Step 3: MITRE ATT&CK Mapping

  • T1059.001: Command and Scripting Interpreter: PowerShell
  • T1105: Ingress Tool Transfer (potential malware download)
  • T1546.011: Event Triggered Execution

Step 4: Recommended Actions

  1. Investigate: What was the encoded PowerShell command?
  2. Check: Were any files downloaded to the endpoint?
  3. Verify: Is this legitimate admin activity or compromise?
  4. Monitor: Watch for additional suspicious processes
  5. Document: Record findings for potential incident
